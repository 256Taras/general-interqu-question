# Міграції та ребалансування — Питання для інтерв'ю

## Жива міграція даних без даунтайму

```
"Нам потрібно змінити стратегію шардування з hash-based на range-based.
Як це зробити без даунтайму?"
```

### Підхід: поетапна міграція (Dual-Write + Switch)

```
Фаза 1        Фаза 2         Фаза 3         Фаза 4
Dual-Write → Data Backfill → Switch Read → Cleanup

Старий ────────────────────────────────── Вимкнення
Новий  ────────────────────────────────── Основний
```

---

### Фаза 1: Dual-Write (паралельний запис)

```
Всі ЗАПИСИ йдуть в обидва кластери (старий і новий)
Всі ЧИТАННЯ йдуть зі старого кластера

Client → Write Proxy → Old Cluster (hash-based) ← reads
                      → New Cluster (range-based)
```

**Що робимо:**
```
1. Розгортаємо новий кластер з range-based sharding
2. Модифікуємо write path — записуємо в обидва
3. Всі нові дані потрапляють в обидва кластери
4. Читаємо ТІЛЬКИ зі старого (поки)
```

**Важливо:**
- Записи в новий кластер асинхронні (не блокують основний flow)
- Помилки запису в новий кластер логуються, але не впливають на клієнтів
- Моніторинг: порівнюємо кількість записів в обох кластерах

---

### Фаза 2: Data Backfill (копіювання історичних даних)

```
Копіюємо дані зі старого кластера в новий (batch процес)

Old Cluster → Backfill Job → New Cluster
              (batch reads)   (batch writes)
```

**Як:**
```python
async def backfill():
    last_processed_id = get_checkpoint()

    while True:
        # Читаємо батч зі старого кластера
        batch = await old_cluster.read_batch(
            after_id=last_processed_id,
            limit=1000
        )
        if not batch:
            break

        # Записуємо в новий кластер (upsert — ідемпотентно)
        await new_cluster.upsert_batch(batch)

        # Зберігаємо checkpoint (для resume при failure)
        save_checkpoint(batch[-1].id)
        last_processed_id = batch[-1].id
```

**Важливо:**
- Використовуємо `upsert` (не `insert`) — дані з Фази 1 вже можуть бути в новому кластері
- Checkpoint для resume при збоях
- Throttling — не перевантажуємо старий кластер читаннями
- Запускаємо в off-peak hours

---

### Фаза 3: Switch Read (переключення читань)

```
Поступово переключаємо читання на новий кластер

Крок 3a: Верифікація даних
- Порівнюємо випадкові записи між кластерами
- Переконуємось що всі дані мігровані

Крок 3b: Shadow reads (canary)
- Читаємо з обох, порівнюємо результати
- Відповідаємо зі старого, логуємо різницю

Крок 3c: Поступове переключення (feature flag)
- 1% трафіку → новий кластер
- 10% → 25% → 50% → 100%
- Моніторимо latency, errors, data correctness
```

```python
async def read_user(user_id):
    if feature_flag.percentage("new_cluster_reads") > random():
        # Читаємо з нового кластера
        return await new_cluster.read(user_id)
    else:
        # Читаємо зі старого кластера
        return await old_cluster.read(user_id)
```

---

### Фаза 4: Cleanup (завершення)

```
1. Вимикаємо dual-write (пишемо тільки в новий)
2. Зупиняємо старий кластер
3. Видаляємо feature flags
4. Архівуємо старий кластер (backup на всяк випадок)
```

**Rollback plan:**
```
На кожній фазі — можливість повернутись назад:
Фаза 1: Вимикаємо dual-write → старий кластер працює як раніше
Фаза 2: Зупиняємо backfill → нічого не зламано
Фаза 3: Переключаємо feature flag на 0% → читання зі старого
Фаза 4: Після видалення старого кластера → rollback вже неможливий
         (тому тримаємо backup)
```

---

### Таймлайн типової міграції

```
Тиждень 1: Підготовка нового кластера, тестування на staging
Тиждень 2: Фаза 1 — включаємо dual-write
Тиждень 3-4: Фаза 2 — backfill (залежить від об'єму даних)
Тиждень 5: Фаза 3 — shadow reads + поступове переключення
Тиждень 6: Фаза 4 — cleanup, моніторинг
```

## Ребалансування при додаванні/видаленні шардів

```
"У нас 4 шарди і ми хочемо додати ще 2. Як перемістити дані
без даунтайму?"
```

### З Consistent Hashing

```
Додаємо shard5 і shard6 на кільце:

До:   [shard1, shard2, shard3, shard4] — рівномірно на кільці
Після: [shard1, shard2, shard3, shard4, shard5, shard6]

Тільки ~1/3 даних потрібно перемістити (ключі, які тепер належать новим шардам)
```

### Процес ребалансування

```
1. Додаємо новий шард у кластер (порожній)
2. Обчислюємо які ключі мають переїхати
3. Починаємо копіювання даних (background)
4. Оновлюємо routing table (поступово)
5. Видаляємо скопійовані дані зі старого шарда
```

**Під час ребалансування:**
```
Запис: йде на НОВИЙ шард (за оновленим routing)
Читання:
  - Спочатку перевіряємо новий шард
  - Якщо не знайдено — fallback на старий (дані ще не перенесені)
```

```python
async def read_key(key):
    new_shard = routing.get_new_shard(key)
    old_shard = routing.get_old_shard(key)

    # Спочатку пробуємо новий шард
    result = await new_shard.get(key)
    if result:
        return result

    # Fallback на старий (дані ще мігруються)
    result = await old_shard.get(key)
    if result:
        # Lazy migration: копіюємо на новий шард при читанні
        await new_shard.set(key, result)
        return result

    return None
```

### Стратегії мінімізації впливу

| Стратегія | Опис |
|---|---|
| Background copy | Копіювання в фоні з throttling |
| Lazy migration | Переміщення при першому зверненні до ключа |
| Hybrid | Background для cold data + lazy для hot data |
| Chunk-based | Переміщення блоками, з паузами між блоками |

## Зміна shard key без даунтайму

```
"Ми шардували по user_id, але тепер потрібно шардувати по tenant_id
(додали multi-tenancy). Як мігрувати?"
```

### Підхід: Double-Indexing

```
Фаза 1: Додаємо tenant_id до всіх записів (backfill)

Фаза 2: Dual-write з обома ключами
  Старий шардинг: hash(user_id) % N
  Новий шардинг: hash(tenant_id) % N
  → Записуємо в обидва

Фаза 3: Backfill існуючих даних в нову схему

Фаза 4: Переключаємо reads на нову схему

Фаза 5: Вимикаємо стару схему
```

**Ключова різниця від простої міграції:** shard key змінився, тому ВСІ дані переміщуються (не частина як при додаванні шарда). Це довше і потребує більше ресурсів.
