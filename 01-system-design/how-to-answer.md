# Як відповідати на System Design питання

## Методологія (Framework)

На інтерв'ю у тебе зазвичай 35-45 хвилин на одне system design питання. Ось перевірений framework:

### Крок 1: Уточнення вимог (3-5 хвилин)

**Не починай проектувати одразу!** Задай питання:

**Функціональні вимоги:**
- Які основні use cases?
- Хто користувачі? (B2B, B2C, internal)
- Які дії може виконувати користувач?
- Чи потрібен real-time?

**Нефункціональні вимоги:**
- Скільки користувачів? (DAU — Daily Active Users)
- Яке навантаження? (reads vs writes ratio)
- Latency вимоги? (< 200ms, < 1s)
- Availability вимоги? (99.9%? 99.99%?)
- Consistency чи Availability важливіше?
- Географія користувачів?

**Приклад:** "Перш ніж проектувати, хочу уточнити кілька речей. Скільки приблизно DAU ми очікуємо? Які основні операції будуть найчастішими?"

---

### Крок 2: Оцінка масштабу (3-5 хвилин)

**Back-of-the-envelope calculations:**

```
Приклад для Twitter:
- 300M MAU, 50% DAU = 150M DAU
- Кожен читає 100 tweets/день = 15B reads/день
- 10% пишуть 2 tweets/день = 30M writes/день

QPS (Queries Per Second):
- Read QPS: 15B / 86400 ≈ 175K QPS
- Write QPS: 30M / 86400 ≈ 350 QPS
- Peak (3x): Read ~500K QPS, Write ~1K QPS

Storage:
- Tweet: ~300 bytes (text) + metadata
- 30M × 300 bytes × 365 days = ~3TB/рік (тільки текст)
- + медіа: значно більше

Bandwidth:
- Read: 175K × 300 bytes ≈ 50 MB/s
- + медіа: 175K × 1MB (avg) ≈ 175 GB/s (потрібен CDN)
```

**Корисні числа для запам'ятовування:**
- 1 день = 86,400 секунд ≈ 100K секунд
- 1 місяць ≈ 2.5M секунд
- 1 рік ≈ 30M секунд
- 1 QPS = ~2.5M запитів/місяць
- 1 MB/s ≈ 2.5TB/місяць

---

### Крок 3: High-Level дизайн (10-15 хвилин)

Намалюй основні компоненти:

```
Клієнт → CDN (статика)
       → Load Balancer → API Servers → Cache (Redis)
                                     → Database
                                     → Message Queue → Workers
                                     → Object Storage (S3)
```

**Для кожного компонента поясни:**
- Чому він потрібен
- Який протокол/технологію обрав і чому
- Як дані flow через систему

**Важливо:** покажи основний read та write flow.

---

### Крок 4: Деталізація компонентів (10-15 хвилин)

Інтерв'юер попросить заглибитись у конкретні частини. Будь готовий:

**Database design:**
- Які таблиці/колекції?
- Які індекси?
- SQL чи NoSQL і чому?
- Partitioning/Sharding стратегія?

**API design:**
```
POST /api/v1/tweets
  Body: { content: string, media?: string[] }
  Response: { id, content, createdAt }

GET /api/v1/feed?cursor=xxx&limit=20
  Response: { data: Tweet[], nextCursor: string }
```

**Caching:**
- Що кешуємо?
- Яка стратегія (cache-aside, write-through)?
- TTL?
- Invalidation?

---

### Крок 5: Масштабування та оптимізація (5-10 хвилин)

- Horizontal scaling для API серверів
- Database replication та sharding
- CDN для статичних ресурсів
- Async processing через message queues
- Caching на різних рівнях

---

### Крок 6: Обробка помилок та моніторинг (2-3 хвилини)

- Як система поводиться при збої компонента?
- Circuit breakers
- Retry з backoff
- Graceful degradation
- Monitoring: latency, error rate, throughput
- Alerting

---

## Типові помилки на інтерв'ю

### ❌ Помилка 1: Одразу починати проектувати
Без уточнення вимог ти можеш спроектувати не те, що потрібно.
**✅ Рішення:** Завжди починай з питань.

### ❌ Помилка 2: Мовчки думати
Інтерв'юер не може оцінити те, що не чує.
**✅ Рішення:** Думай вголос. "Я розглядаю два варіанти..."

### ❌ Помилка 3: Заглиблюватись у деталі занадто рано
Не витрачай 20 хвилин на database schema, якщо ще немає high-level design.
**✅ Рішення:** Спершу загальна картина, потім деталі.

### ❌ Помилка 4: Не обговорювати trade-offs
Кожне рішення має плюси і мінуси.
**✅ Рішення:** "Я обрав Redis тому що потрібен low-latency доступ, але це означає що ми маємо eventual consistency."

### ❌ Помилка 5: Використовувати технології без розуміння
"Давайте використаємо Kafka" — чому саме Kafka, а не RabbitMQ?
**✅ Рішення:** Обґрунтовуй кожен вибір технології.

### ❌ Помилка 6: Ігнорувати масштаб
Проект для 1000 користувачів і для 100M — це різні системи.
**✅ Рішення:** Масштаб визначає архітектуру.

### ❌ Помилка 7: Over-engineering
Не потрібен Kubernetes для MVP на 1000 юзерів.
**✅ Рішення:** Починай просто, масштабуй по потребі.

## Шаблон відповіді

"Дякую за питання. Перш ніж почати проектувати, хочу уточнити кілька вимог...

[Питання та уточнення]

Добре, тепер оцінимо масштаб...

[Розрахунки]

Починаю з high-level архітектури. На верхньому рівні система складається з...

[Малюнок + пояснення]

Тепер поглибимось у ключові компоненти...

[Деталізація]

Щодо масштабування, я б запропонував...

[Оптимізація]

Щодо відмовостійкості...

[Error handling, monitoring]"

## Корисні фрази

- "Це залежить від вимог, але за замовчуванням я б обрав..."
- "Trade-off тут такий: ми отримуємо X, але втрачаємо Y"
- "Для MVP достатньо простого підходу, але якщо масштаб зросте..."
- "Я бачу два підходи. Перший... Другий... Я б обрав перший, тому що..."
- "Bottleneck тут може бути в..., тому я додаю..."
- "Цей компонент може стати single point of failure, тому..."
