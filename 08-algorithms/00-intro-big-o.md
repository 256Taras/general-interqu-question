# Вступ: що таке алгоритм і як міряти його ефективність (Big O)

## Що таке алгоритм? Навіщо взагалі це знати на співбесіді?

**Алгоритм** -- це покрокова інструкція для розв'язання задачі. Наприклад: "знайти максимальне число в масиві" -- це задача, а конкретні кроки `for (let x of arr) if (x > max) max = x` -- це алгоритм.

**Структура даних** -- спосіб зберігання даних, оптимізований під певні операції. Масив, об'єкт, Map, Set -- це структури даних. Вибір структури часто визначає, наскільки швидким буде алгоритм.

**Навіщо на співбесіді?**
1. На співбесідах у FAANG, великих стартапах, fintech -- є алгоритмічна секція. Без неї оффер не отримаєш.
2. Уміння думати про ефективність коду впливає на щоденну роботу: чому цей endpoint повільний? Чому сервер падає на 10k users? Часто відповідь -- у базовій алгоритміці.
3. Це про структурне мислення: як розбити велику задачу на менші підзадачі.

**Головна ідея:** інтерв'юер не хоче бачити, що ти пам'ятаєш алгоритми напам'ять. Він хоче бачити, як ти **думаєш** -- як розбиваєш задачу, як думаєш про граничні випадки, як оцінюєш swап між часом і пам'яттю.

## Що таке Big O і чому треба почати з цього?

Big O -- це **спосіб описати, як швидкість (або споживання пам'яті) алгоритму зростає зі збільшенням розміру вхідних даних**.

Уяви дві функції для пошуку числа у списку:

```javascript
// Варіант A: лінійний пошук
function findA(arr, target) {
  for (let i = 0; i < arr.length; i++) {
    if (arr[i] === target) return i;
  }
  return -1;
}

// Варіант B: бінарний пошук (масив уже відсортований)
function findB(sortedArr, target) {
  let left = 0, right = sortedArr.length - 1;
  while (left <= right) {
    const mid = Math.floor((left + right) / 2);
    if (sortedArr[mid] === target) return mid;
    if (sortedArr[mid] < target) left = mid + 1;
    else right = mid - 1;
  }
  return -1;
}
```

На масиві з 10 елементів різниця невелика. На масиві з 1 мільярда:
- `findA` в найгіршому випадку перевірить мільярд елементів.
- `findB` -- максимум 30 перевірок (`log₂(1_000_000_000) ≈ 30`).

Big O дозволяє це виразити формально:
- `findA` -- **O(n)** (час росте лінійно з розміром).
- `findB` -- **O(log n)** (час росте логарифмічно).

```
    час виконання
     │
     │         O(n²)
     │       /
     │      /
     │     /         O(n)
     │    /        /
     │   /       /
     │  /     /
     │ /   /      O(log n)
     │/__/_______________________    O(1)
     └─────────────────────────── n (розмір вхідних даних)
```

## Формальне визначення та правила спрощення

**Big O** описує асимптотичну верхню межу. Тобто: "при дуже великих n, час виконання не перевищує цю функцію, помножену на якусь константу".

Формально: `f(n) = O(g(n))`, якщо існують такі константи `c > 0` і `n₀`, що `f(n) ≤ c × g(n)` для всіх `n ≥ n₀`.

**Це не треба пам'ятати.** Треба пам'ятати **три практичні правила**:

**Правило 1: Викидай константи.**
```javascript
function example(arr) {
  for (let x of arr) { /* робота */ }    // n операцій
  for (let x of arr) { /* інша робота */ }   // + n операцій
  for (let x of arr) { /* ще робота */ }     // + n операцій
}
// Разом: 3n операцій → O(n), а не O(3n)
```

Константа не має значення для асимптотики. `1000n` і `2n` -- обидва O(n).

**Правило 2: Викидай менші члени.**
```javascript
function example(arr) {
  for (let i = 0; i < arr.length; i++) {       // n
    for (let j = 0; j < arr.length; j++) {     // n × n = n²
      /* робота */
    }
  }

  for (let x of arr) { /* робота */ }           // + n
}
// Разом: n² + n → O(n²), бо n² домінує при великих n
```

**Правило 3: Різні входи -- різні змінні.**
```javascript
function compare(arrA, arrB) {
  for (let a of arrA) {       // |A| операцій
    for (let b of arrB) {     // |B| операцій
      /* робота */
    }
  }
}
// Це O(a × b), де a = arrA.length, b = arrB.length.
// НЕ можна писати O(n²), бо a і b -- різні.
```

## Головні класи складності (вивчи їх напам'ять)

| Big O         | Назва            | Приклад                                           | n=10      | n=1M          |
|---------------|------------------|---------------------------------------------------|-----------|----------------|
| O(1)          | Константа        | Доступ до елемента за індексом `arr[5]`          | 1         | 1              |
| O(log n)      | Логарифмічна     | Бінарний пошук                                    | ~3        | ~20            |
| O(n)          | Лінійна          | Обхід масиву `for of`                             | 10        | 1,000,000      |
| O(n log n)    | Лінійно-лог.     | Ефективне сортування (merge, quicksort average)   | ~33       | ~20,000,000    |
| O(n²)         | Квадратична      | Вкладені цикли, bubble sort                       | 100       | 10¹²           |
| O(n³)         | Кубічна          | Три вкладені цикли                                | 1,000     | 10¹⁸           |
| O(2ⁿ)         | Експоненційна    | Рекурсивне обчислення Фібоначчі без мемоізації    | 1024      | незбагненно    |
| O(n!)         | Факторіальна     | Перебір всіх перестановок                          | 3.6M      | ніколи         |

**Візуальне правило:** якщо бачиш два вкладених цикли по тому самому масиву -- O(n²). Якщо один цикл -- O(n). Якщо на кожному кроці ділиш задачу навпіл -- O(log n).

**Правило співбесіди:** O(n²) на масиві з n=10⁶ не пройде таймаут. O(n log n) -- прийнятно. O(n) -- ідеально.

## Як рахувати Big O на практиці (кроки)

**1. Порахуй, скільки разів виконується найглибший цикл.**

```javascript
function sumPairs(arr) {
  let total = 0;
  for (let i = 0; i < arr.length; i++) {        // ← зовнішній: n разів
    for (let j = i + 1; j < arr.length; j++) {  // ← внутрішній: у середньому n/2
      total += arr[i] + arr[j];                  // ← робота O(1)
    }
  }
  return total;
}
// n × n/2 × O(1) = n²/2 → O(n²)
```

Хоча внутрішній цикл стартує з `i+1`, а не з 0 -- це все одно O(n²). Правило: викидай константи (1/2 тут).

**2. Додавай складності послідовних блоків, бери найбільшу.**

```javascript
function process(arr) {
  arr.sort();                                   // ← O(n log n)
  for (let x of arr) { /* робота */ }           // ← O(n)
  return arr[0];                                // ← O(1)
}
// O(n log n) + O(n) + O(1) → O(n log n)
```

**3. Для рекурсії -- порахуй кількість викликів і роботу на кожному.**

```javascript
function fibonacci(n) {
  if (n <= 1) return n;
  return fibonacci(n - 1) + fibonacci(n - 2);
}
```

Дерево викликів:
```
                   fib(5)
                  /      \
             fib(4)       fib(3)
             /    \       /    \
         fib(3) fib(2) fib(2) fib(1)
         /  \    /\    /\
       ...      ...
```

На кожному рівні кількість викликів подвоюється. Глибина -- n. Тому O(2ⁿ). Це катастрофа -- `fibonacci(50)` виконуватиметься годинами.

**4. Для скороченого пошуку/поділу -- зазвичай log.**

```javascript
while (n > 1) {
  n = Math.floor(n / 2);       // кожна ітерація ділить n навпіл
}
// Кількість ітерацій: log₂(n) → O(log n)
```

## Best case, Average case, Worst case

Часто алгоритм поводиться по-різному залежно від вхідних даних.

**Приклад: лінійний пошук у масиві.**
- **Best case** -- шуканий елемент перший: O(1).
- **Worst case** -- шуканий елемент останній або відсутній: O(n).
- **Average case** -- випадкове положення: приблизно n/2, тобто O(n).

На співбесіді зазвичай говорять про **worst case** (Big O). Іноді -- average (Θ або тета). Best case (Ω) рідко цікавий.

**Quicksort** -- класичний приклад:
- Best/Average: O(n log n).
- Worst case: O(n²) (коли масив вже відсортований і ти завжди вибираєш поганий pivot).

## Space complexity: пам'ять теж важлива

Big O описує не тільки час, а й пам'ять.

```javascript
// O(1) space -- не створюємо додаткових структур пропорційно до n
function sum(arr) {
  let total = 0;                 // 1 змінна
  for (let x of arr) total += x;
  return total;
}

// O(n) space -- створюємо новий масив розміром n
function double(arr) {
  const result = [];
  for (let x of arr) result.push(x * 2);    // масив росте до n
  return result;
}

// O(n) space -- рекурсія теж займає stack
function recursiveSum(arr, i = 0) {
  if (i === arr.length) return 0;
  return arr[i] + recursiveSum(arr, i + 1);
  // Глибина стеку: n викликів → O(n) space
}
```

**Typical trade-off:** можна використати більше пам'яті, щоб прискорити алгоритм. Наприклад, додати Map для кешу (O(n) space) і зробити пошук O(1) замість O(log n).

На співбесіді завжди проговорюй і time, і space complexity.

## Amortized complexity: коли буває "в середньому швидко"

Деякі операції зрідка коштують дорого, але в середньому дешеві.

**Приклад: додавання в кінець масиву (JavaScript `push`).**

Під капотом масив -- це неперервний блок пам'яті. Коли він заповнюється, двигун JS виділяє новий блок вдвічі більший і копіює всі елементи.

- Якщо є місце -- `push` за O(1).
- Якщо немає -- O(n) для копіювання.

Але якщо ти робиш n `push`-ів:
- O(n) робота на копіювання трапляється раз на n операцій.
- Сумарна робота: n × O(1) + O(n) копіювань ≈ 2n = O(n).
- На **кожну окрему операцію** в середньому приходиться O(1).

Це **amortized O(1)** -- "в середньому O(1), хоча іноді буває дорожче".

Hash map (об'єкти, Map у JS) так само -- `set/get` amortized O(1), але з rehash при зростанні.

## Big O у стандартних структурах JavaScript

Це must-know для співбесіди. Запам'ятай або хоча б зрозумій логіку.

### Array

| Операція               | Big O      | Коментар                              |
|------------------------|------------|----------------------------------------|
| `arr[i]` (доступ)      | O(1)       | Пряме звернення по індексу             |
| `arr.length`           | O(1)       |                                        |
| `arr.push(x)`          | O(1) amort.| Іноді O(n) через reallocation          |
| `arr.pop()`            | O(1)       |                                        |
| `arr.unshift(x)`       | O(n)       | **Зсуває всі елементи!**               |
| `arr.shift()`          | O(n)       | Те саме                                |
| `arr.includes(x)`      | O(n)       | Лінійний пошук                         |
| `arr.indexOf(x)`       | O(n)       |                                        |
| `arr.find(fn)`         | O(n)       |                                        |
| `arr.slice(a, b)`      | O(b - a)   | Копіює піддіапазон                     |
| `arr.splice(i, k)`     | O(n)       | Зсуває елементи після i                |
| `arr.concat(other)`    | O(n + m)   |                                        |
| `arr.sort()`           | O(n log n) | Зазвичай TimSort в V8                  |
| `arr.reverse()`        | O(n)       |                                        |
| `[...arr]`, `arr.map`  | O(n)       |                                        |

**Важливо:** `unshift` і `shift` -- O(n). Якщо потрібна структура з швидкими операціями на початку -- використовуй Deque (двобічну чергу) або уникай цього.

### Object / Map / Set

| Операція                | Big O (avg)  | Коментар                         |
|-------------------------|--------------|----------------------------------|
| `map.get(key)`          | O(1)         | Hash lookup                       |
| `map.set(key, val)`     | O(1) amort.  | Rehash при зростанні             |
| `map.has(key)`          | O(1)         |                                  |
| `map.delete(key)`       | O(1)         |                                  |
| `map.size`              | O(1)         |                                  |
| `obj[key]`              | O(1) avg     | Як `map.get`, але повільніше     |
| `set.add(x)`            | O(1) amort.  |                                  |
| `set.has(x)`            | O(1)         |                                  |

**Map vs Object:**
- `Map` -- справжня hash map. Розроблена для частих операцій, зберігає порядок вставки.
- `Object` -- теж hash map, але зі своїми quirks (prototype chain, ключі тільки string/Symbol).
- Для алгоритмічних задач **завжди використовуй `Map` і `Set`**.

### String

```javascript
str[i]         // O(1)
str.length     // O(1)
str + other    // O(n + m) -- створює новий рядок
str.indexOf(x) // O(n × m) у найгіршому випадку
str.slice(a,b) // O(b - a)
str.split(x)   // O(n)
```

**Важливо:** рядки в JS immutable. `str += "x"` у циклі створює O(n²) роботи, бо щораз копіює весь рядок. У таких випадках:
```javascript
// ❌ Погано: O(n²) через immutability рядка
let result = "";
for (let x of arr) result += x;

// ✅ Добре: O(n)
const parts = [];
for (let x of arr) parts.push(x);
const result = parts.join("");
```

## Типові пастки при оцінці Big O

**1. Приховані цикли в стандартних методах.**

```javascript
function hasDuplicate(arr) {
  for (let i = 0; i < arr.length; i++) {
    if (arr.includes(arr[i], i + 1)) return true;  // includes -- це O(n)!
  }
  return false;
}
// Це O(n²), а не O(n)
```

**2. `indexOf` / `includes` всередині циклу.**

Якщо бачиш `arr.includes(x)` у циклі -- це червоний прапорець. Замінюй на `Set`.

```javascript
// ❌ O(n²)
function intersection(a, b) {
  return a.filter(x => b.includes(x));
}

// ✅ O(n + m)
function intersection(a, b) {
  const setB = new Set(b);
  return a.filter(x => setB.has(x));
}
```

**3. Рекурсія без мемоізації -- часто експоненційна.**

Дивись приклад з Fibonacci вище.

**4. Concatenation рядка в циклі -- O(n²).**

Див. приклад з `result += x`.

**5. Spread operator -- O(n).**

```javascript
function add(arr, x) {
  return [...arr, x];   // O(n) -- копіює весь масив
}

// Якщо це в циклі -- O(n²)
```

## Як на співбесіді проговорювати Big O

Порядок:
1. **Спочатку сформулюй задачу своїми словами.** "Отже, у мене масив чисел і ціль, треба знайти індекси двох чисел, що дають цю ціль. Масив може бути великий -- до 10⁵."
2. **Запропонуй наївне рішення і озвуч його складність.** "Найпростіше -- два вкладених цикли, перевіряємо всі пари. Це O(n²) time, O(1) space."
3. **Запитай, чи це прийнятно.** Часто інтерв'юер скаже "можеш краще?" -- це сигнал, що треба шукати оптимізацію.
4. **Запропонуй оптимізоване рішення.** "Можу зробити O(n) time, використовуючи Map: іду по масиву один раз, для кожного x перевіряю, чи target - x вже є у Map. Space стає O(n)."
5. **Проговори trade-off.** "Витратив пам'ять заради швидкості. Для n=10⁵ це прийнятно."

Інтерв'юери люблять, коли кандидат **сам** піднімає питання складності, не чекаючи на навідні запитання.

## Типові складності для класичних задач (шпаргалка)

- **Пошук у відсортованому масиві** → O(log n) бінарним пошуком.
- **Пошук у невідсортованому** → O(n).
- **Сортування** → O(n log n) ефективним алгоритмом.
- **Обхід дерева/графа** → O(V + E) або O(n) для дерева.
- **Знайти унікальні елементи** → O(n) через Set.
- **Знайти найчастіший елемент** → O(n) через Map.
- **Перемножити всі пари** → O(n²).
- **Згенерувати всі підмножини** → O(2ⁿ).
- **Згенерувати всі перестановки** → O(n!).

## Мінімальний чекліст перед тим як писати код

Задача -> вдаси цю послідовність:

1. **Розмір вхідних даних.** n до 10? 10⁴? 10⁹? Визначає максимально допустиму складність.
2. **Типові розміри і допустима складність** (для 1 секунди на звичайному сервері):
   - n ≤ 10: будь-що, включно з O(n!).
   - n ≤ 20: O(2ⁿ).
   - n ≤ 500: O(n³).
   - n ≤ 5000: O(n²).
   - n ≤ 10⁶: O(n log n) або O(n).
   - n ≤ 10⁹: O(log n) або O(1).
3. **Структура, яку вже маєш.** Масив, об'єкт, дерево, граф?
4. **Граничні випадки.** Порожній вхід, один елемент, дублікати, від'ємні числа.
5. **Мета оптимізації.** Час, пам'ять, чи обидва?

## Практичні приклади: проаналізуй сам

Подивись на цей код і скажи Big O, перш ніж читати відповіді.

### Приклад 1
```javascript
function f(arr) {
  let sum = 0;
  for (let x of arr) sum += x;
  return sum;
}
```
<details><summary>Відповідь</summary>O(n) time, O(1) space.</details>

### Приклад 2
```javascript
function f(arr) {
  for (let i = 0; i < arr.length; i++) {
    for (let j = 0; j < arr.length; j++) {
      console.log(arr[i], arr[j]);
    }
  }
}
```
<details><summary>Відповідь</summary>O(n²) time, O(1) space.</details>

### Приклад 3
```javascript
function f(n) {
  if (n <= 1) return 1;
  return f(n - 1) * n;
}
```
<details><summary>Відповідь</summary>O(n) time (n викликів), O(n) space (глибина стеку).</details>

### Приклад 4
```javascript
function f(arr) {
  arr.sort();
  for (let x of arr) console.log(x);
}
```
<details><summary>Відповідь</summary>O(n log n) time (sort домінує), O(1) або O(n) space залежно від реалізації sort.</details>

### Приклад 5
```javascript
function f(a, b) {
  for (let x of a) {
    for (let y of b) {
      console.log(x, y);
    }
  }
}
```
<details><summary>Відповідь</summary>O(a × b) time, O(1) space. НЕ O(n²), бо a і b різні.</details>

### Приклад 6
```javascript
function f(n) {
  let i = 1;
  while (i < n) {
    i *= 2;
  }
}
```
<details><summary>Відповідь</summary>O(log n) time. Кожна ітерація подвоює i -- класична логарифмічна поведінка.</details>

## Ключові думки

- Big O -- мова спілкування про ефективність. Без неї не можна обговорювати оптимізацію.
- Викидай константи, викидай менші члени. Ти описуєш ріст, не точний час.
- Завжди проговорюй **time AND space** на співбесіді.
- Знай стандартні складності JS-операцій напам'ять.
- `unshift`, `shift`, `splice`, `includes` -- червоні прапорці в циклі.
- Набір "за замовчуванням" для оптимізації: Map, Set для O(1) пошуку; сортування для O(n log n) структурування; бінарний пошук для O(log n).
