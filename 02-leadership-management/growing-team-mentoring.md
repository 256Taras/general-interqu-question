# Менторство та Розвиток Команди

## Що таке менторство і відмінність від coaching / teaching?

**Менторство** — це довгострокові стосунки, де досвідчена особа (ментор) передає знання, навички та мудрість молодшому розробнику. Ментор діагностує прогалини, підказує напрямок, але не дає готові рішення.

**Coaching** — це цільована допомога для досягнення конкретного результату. Coach допомагає прояснити цілі, визначити перешкоди й розробити дійсний план. Коротше за часом ніж менторство.

**Teaching** — пряма передача знань. Вчитель пояснює концепцію, показує приклади, переконується в розумінні. Менш діалогово.

```
Менторство:      1-2 роки    │ Довгострокові відносини    │ Передача досвіду
Coaching:        3-6 місяців │ Цільовані задачі            │ Допомога у досягненні
Teaching:        1-3 тижні   │ Передача специфічного      │ Специфічне знання
```

**На співбесіді скажи:**
"Я працюю як ментор: я спостерігаю за junior-ом, визначаю його слабкі місця, намагаюся дати простір для навчання через помилки, але не кидаю його на самоту. Я не даю готові рішення — спочатку питаю 'як би ти це розв'язав?', а потім керую до правильного напрямку".

## Як ростити junior-а: онбординг, перші 30-60-90 днів

### День 1-2: Основи інфраструктури

- Постав на машину: IDE, Node.js, Git, Docker
- Покажи репозиторій, dev процес, як запустити сервер локально
- Введи в команду: хто-це-все, де живе документація, як писати PR
- Дай простий task: fix typo, додай коментар до функції, оновлення README
- Люди завжди нервні на початку — будь дружелюбний, повільний

```javascript
// День 1: Попроси юніора зробити щось дуже простеньке
// Наприклад, додати логування в функцію:

// BEFORE:
function getUserById(id) {
  return db.query(`SELECT * FROM users WHERE id = ?`, [id]);
}

// AFTER: Junior додав:
function getUserById(id) {
  console.log(`Fetching user ${id}`);
  return db.query(`SELECT * FROM users WHERE id = ?`, [id]);
}

// Це переконує його що:
// 1. Код можна змінювати без страху
// 2. Він вже контрибьютив в проект
// 3. Code review процес працює й він вже долучився
```

### Місяці 1-3: Рамка адаптації

**Тиждень 1-2 (Розуміння архітектури)**
- Покажи основні модулі: API layer, DB layer, business logic
- Поясни flow запиту від входу до виходу
- Дай читати архітектуру, ale не всю відразу

**Тиждень 3-4 (Перші реальні task-и)**
- Одна задача = один файл, одна функція (не складна логіка)
- Проблеми: додавання нового поля в базу, fix простого бага
- Юніор повинен йти до тебе з питаннями — це норма

**Місяць 2-3 (Незалежність зростає)**
- Задачі важчають: від одного файлу до цілої фічі
- Юніор починає давати code review іншим (з наглядом)
- Приблизно 60-70% часу він працює самостійно, 30% на questions/pairing

### Red flags, на які дивитися:

1. Юніор сидить тихо кілька днів без питань → виглядає, як він бовтається
2. Юніор задає САМО питання → це добре
3. Юніор не може запустити сервер після 1 тижня → проблема з онбордингом

## 1:1 meetings — структура, частота, ідеї тем

### Частота й тривалість

- Junior: 1 раз на тиждень, 30-40 хвилин
- Mid-level: 1 раз на 2 тижні, 20-30 хвилин
- Senior: 1 раз на місяць, 30 хвилин (більше про стратегію)

### Структура 1:1

```
ПОЧАТОК (2 хв)
├─ Як справи? (лід-ін, людський тон)

СЕРЕДИНА (20-25 хв)
├─ Розбір завдань
├─ Технічні питання (якщо є)
├─ Знайти pain points
├─ Career goals

КІНЕЦЬ (5 хв)
├─ Action items (3 максимум)
├─ Наступна зустріч
└─ Позитивне завершення
```

### Теми, які варто піднімати:

**Для junior-а:**
- "Які теми в codesbase тебе найбільше збивають з толку?"
- "Хто найбільше допомагає тобі в команді? (Ментор для соціалізації)"
- "Крок за кроком: як ти пишеш PR? Що тебе складає?"
- "Які ерори ти повторюєш?"

**Для mid-level:**
- "Над якими навичками ти хочеш працювати?"
- "Де ти бачиш себе через 2 роки?"
- "Чи є task-и, які тебе затримують? Як ми можемо їх розділити?"

**Для senior:**
- "Які стратегічні цілі тебе цікавлять? (система дизайну, перфоманс, тesting)"
- "Хто в команді потребує більше менторства від тебе?"

## Code review як навчальний інструмент

### Правила конструктивного review

**1. Питай замість команди**
```javascript
// ПОГАНО (команда):
// "Це неправильно, ми не робимо так"

// ДОБРЕ (питання):
// "Чому ти вибрав цей підхід? Чи розглядав ти Promise.all() тут?"
```

**2. Розділи коментарі на категорії**
- 🔴 **Critical**: безпека, memory leak, race condition
- 🟠 **Important**: архітектура, performance
- 🟡 **Minor**: очікування коду, стиль
- 🟢 **Nice to have**: ідеї для майбутнього

```javascript
// PR comment:

// 🔴 CRITICAL
// Тут є race condition. Між читанням з кешу й записом може змінитися стан.
// Потрібно додати lock.

// 🟠 IMPORTANT
// Performance: ти проходиш масив 3 рази. Можна це зробити в один pass.

// 🟡 MINOR
// Залишилась закоментована строчка в 42 лінії.

// 🟢 NICE TO HAVE
// Цікава ідея: коли накопиться experience, розглянь use memo тут.
```

**3. Похвали хороший код**
```javascript
// Хороший коментар у code review:
// "Мені дуже подобається, як ти обробляєш помилки тут.
//  Давав нам експоненціальний backoff для retries."
```

**4. Запроси зміни, не наказуй**
```javascript
// ПОГАНО: "Переписати цю функцію"
// ДОБРЕ:  "Можемо розглянути рефакторинг цієї функції?
//          Вона 50 рядків і важко читається.
//          Ось ідея: витягнути валідацію в окрему функцію."
```

### Що коментувати vs що ні

| Коментуй | Не коментуй |
|----------|-----------|
| Логіка помилок | Стиль (2 vs 4 spaces) |
| Архітектура (структура) | Назви змінних (якщо ясні) |
| Performance issues | Переносування рядків |
| Security concerns | Порядок импортів |
| Missing tests | Поле переносу каретки |

Причина: ESLint, Prettier й IDE беруть на себе стиль-питання. Ти повинен сидіти на логіці й дизайні.

## Як давати feedback: SBI модель

**SBI = Situation – Behavior – Impact**

Це структура, яка робить feedback конкретнішим й менш персональним.

```
Situation (ситуація)    — що трапилось?
Behavior (поведінка)    — що людина зробила?
Impact (влив)           — який результат?
```

### Приклад позитивного feedback

```
СИТУАЦІЯ:
"Вчора на планерці обговорювали, як зменшити час Response Time на /users endpoint."

ПОВЕДІНКА:
"Ти виступив першим з ідеєю кешувати результати. Навіть запропонував Redis 
й TTL 10 хвилин."

ВПЛИВ:
"Це зберегло б нам ~500ms per request. Команда одразу зацікавилась 
й ми почали реалізацію. Ти дав напрямок обговоренню."
```

### Приклад негативного feedback

```
СИТУАЦІЯ:
"На PR з нового auth-модуля було 5 коментарів з питаннями про безпеку."

ПОВЕДІНКА:
"Ти не відповідав на ці коментарі 3 дні. Коли йти клікнув 'approve', 
ти просто merged без обговорення."

ВПЛИВ:
"Тепер у нас в production потенційна уразливість. Код не було перевірено 
должним чином. Це порушує наш стандарт code review."

(Наступне слово — слухати його пояснення)
```

### За схемою SBI:
1. Людина розуміє точку в часі (не "ти завжди")
2. Фокус на дії, не на характері
3. Наслідок зрозумілий й конкретний

## Delegation — коли й як делегувати

### Матриця делегування за рівнем skill + confidence

```
           LOW SKILL → HIGH SKILL
LOW CONF.  │ Coaching         │ Supporting    │
HIGH CONF. │ Mentoring        │ Delegating    │
```

**Coaching** (Низький skill, низька впевненість)
- Близький нагляд, частій check-ins
- Ти робиш більш разом, юніор спостерігає й запитує
- Приклад: написання першого API endpoint

**Supporting** (Низький skill, висока впевненість)
- Юніор впевнений, ale потребує support
- Ти дозволяєш йому вести, але доступний для питань
- Приклад: додавання нового функціоналу до відомого модуля

**Mentoring** (Високий skill, низька впевненість)
- Юніор знає що робити, але сумнівається в собі
- Твоя роль: підтримка й валідація вибору
- Приклад: рефакторинг складного кода після основного роботи

**Delegating** (Високий skill, висока впевненість)
- Повна автономність
- Ти не микро-менеджериш, тільки спостерігаєш результати
- Приклад: senior розробник розвиває новий feature end-to-end

### Pitfalls при делегуванні

**Занадто рано (делегування на рівні "Coaching")**
```
Юніор: "Як написати тесту?"
Ти: "Розбери сам, у мене немає часу"

РЕЗУЛЬТАТ: 
- Тест написаний неправильно
- Юніор хворіє впевненістю
- Технічний борг
```

**Занадто пізно (мікро-менеджмент)**
```
Senior розробник завершив фічу на 80%
Ти: "Хочеш, я переглянув цей PR? Я вважаю там 3 проблеми..."

РЕЗУЛЬТАТ:
- Senior почувається як юніор
- Він не приймає ініціативу в наступний раз
- Bottleneck на тобі
```

### Як правильно делегувати

1. **Визнач рівень: Skill + Confidence**
2. **Дай crystal-clear цільь**: "Додай endpoint /products, повинен повертати JSON із category"
3. **Обговори обмеження**: "Не міняй DB schema без мого дозволу"
4. **Встанови check-ins**: "Покажи мені draft у пятницю"
5. **Слухай (не розповідай) бажані рішення**: "Як би ти це розв'язав?"
6. **Дозволь помилкам трапитись** (якщо не критичні)
7. **Переглянь результат разом**

## Hiring sense — як співбесідувати кандидатів

### Технічна частина: сигнали vs шум

**Шум (не дивись на це):**
- "Ти знаєш React hooks?" → на співбесіді це не дизає
- LeetCode Easy задачі → не відповідає реальній роботі
- "Як вирішити N-queens?" → знання, але не навик

**Сигнали (дивись на це):**
- "Розповідь мені про проект, де ти мав критичний баг. Як ти його знайшов?"
  - Розкриває debugging思 мислення
- "Навіщо ти вибрав монолітну архітектуру замість мікросервісів?"
  - Trade-offs розуміння
- "Як би ти оптимізував цей код? Показуючи на моїх очах"
  - Practical thinking

### Структура tech-інтерв'ю (60 хв)

```
0-5 хв    │ Вступ, привіт, розслаблення кандидата
5-25 хв   │ Система дизайну / архітектура (не кодінг)
25-45 хв  │ Лайв кодінг, але ПРОСТИЙ код + обговорення
45-55 хв   │ Його питання про компанію / проект
55-60 хв   │ Наступні кроки
```

### Приклад архітектурного питання (замість LeetCode)

```
"У нас є мільйон користувачів, і кожен запит до /user/:id займає 500ms.
Як би ти це оптимізував?"

Слухаємо:
- Кешування (Redis) — базовий level
- Читай-тільки реплікація — middle
- Батчинг запросів — senior
- Rate limiting + circuit breaker — думав про stability

Це розкриває реальне мислення.
```

### Behavioral питання

```
"Розповідь про час, коли ти мав conflict у команді з іншим розробником.
Як ви це розв'язали?"

Шукаємо:
✓ Він брав відповідальність (не звинувачував)
✓ Спробував зрозуміти іншого
✓ Запропонував вирішення разом
✓ Навчився чого-небудь

✗ Він виглядає як聖 святий (підозріло)
✗ Весь час був правий, другий помилявся
✗ Не намагався розв'язати - просто escalated to manager
```

### Red flags під час інтерв'ю

1. Не можеш поєднати на папері простий цикл → skill gap
2. Не має питань про проект → не цікаво
3. Постійно говорить про гроші → можливо, не закоханий у роботу
4. Винить попередніх керівників у всьому → культурна проблема

### Green flags

1. Поважає твої питання, витримує паузи для роздумів
2. Питає, як виглядає день-в-день, quale tools вин користуєш
3. Цікавиться людьми в команді, як структура вибудована
4. Бере на себе ошибки, не звинувачує інших

## Знаки burnout у команді й що робити

### Знаки бернаута (дивись на ці поведінки)

**Рівень 1 (Ранні ознаки)**
- Юніор перестав задавати питання (раніше було багато)
- Замість 9-5 зараз 8am-8pm (саме собой)
- Не бере feedback конструктивно, захищається

**Рівень 2 (Помітні зміни)**
- Вже не сміється на stand-up
- Часто відпрацьовує рано, не закінчує задачи
- Pull requests дошлют в 11 вечера в п'ятницю
- Тайм-оути в слаці все менше і менше

**Рівень 3 (Критично)**
- Приходить на 1:1 з червоними очима
- "Я не знаю, чи я хочу писати код..."
- Не фокусується на одному за раз (паніка)

### Що робити

**Одразу (на рівні 1-2):**

1. **Зберегли conversation 1:1**
   ```
   "Я помітив, що останні тижні було напруженим.
    Що трапилось? Як я можу допомогти?"
   ```

2. **Виявити ROOT CAUSE:**
   - Надмір роботи? (розподіліти задачи)
   - Складна архітектура? (привдать більше time на learning)
   - Проблеми в команді? (конфлікт? мікро-менеджмент?)
   - Особисті питання? (личні проблеми часто наїжджають на роботу)

3. **Активні дії:**
   ```javascript
   // Якщо надмір роботи:
   - Заморозити нові фічі на 2 тижні
   - Дати task-и меньше й простіше
   - Більше pairing (зменшити одиночество)

   // Якщо складна архітектура:
   - Організувати knowledge-sharing session
   - Призначити ментора / buddy developer
   - Дати часу на refactoring (не тільки features)

   // Якщо проблеми в команді:
   - Поговорити з особистісно конфліктуючими 1:1
   - Встановити team norms (respect в code review)
   - Показати приклад: як ти даєш feedback конструктивно
   ```

4. **Долгострокові заходи:**
   - Перегляд workload кожні 2 тижні
   - Поговорити з HR про перехід на part-time (якщо потрібно)
   - Гарантовані дні відпустки ("не буди мене в отпуске!")
   - Дати можливість fokusuvat на цікавій технічній роботі

**Red line:**
Якщо після 3-4 тижнів person все одно в critical стані → залучи HR.
Бернаут — це здоров'я, це серйозніше за project deadlines.

## Як побудувати інклюзивну команду

### Психологічна безпека (Amy Edmondson)

Команда з високою психологічною безпекою:
- Говорять свою думку, навіть якщо вона не популярна
- Не бояться запитувати "дурні" питання
- Сказують про помилки одразу, а не приховують
- Пропонують нові ідеї без страху бути пристебленим

Як це виглядає на практиці:

```javascript
// НИЗЬКА психологічна безпека:
Junior на stand-up: "Я закінчив feature"
(Але насправді залишилось 30% роботи — він боїться сказати)

// ВИСОКА психологічна безпека:
Junior на stand-up: "Я закінчив 70% feature.
Заклякнув на сериалізації JSON. 
Буду питати help після stand-up, якщо ок?"
```

### Як створити психологічну безпеку

**1. Бути вразливим першим (як лідер)**
```
"Вчора я дав тобі неправильний advice.
Я подумав, що знаю відповідь, але не перевірив код.
Моя помилка. Спасибо, що виловив."
```

**2. Невірні питання — це нормально**
```
Junior: "Що таке prototype в JavaScript?"
Погане лідер: "Ти не знаєш? Це базовий матеріал!"

Хороший лідер: "Відмінне питання!
 Prototype нелегко зрозуміти на початку.
 Дай мені намалювати на дошці..."
```

**3. Припустити за замовчуванням добру намір**
```
// Натомість:
"Навіщо ти написав цей код так неефективно?"

// Скажи:
"Я вижу, що ти вибрав цей підхід.
Які були причини? Я дивлюся з перспективи 
А чи ми можемо оптимізувати тут?"
```

**4. Публічно визнавати й святкувати помилки**
```
На retro:
"Цього спринту Сергій виловив критичний bug перед продакшеном.
Він мав розпочати pull request і сказати: 'Ей, это looks weird'.
Це ТОЧНО те, що нам потрібно. Добро пожаловать."
```

### Diversity у найму

- Не найманти у себе вдома (bias): запроси знайомих через LinkedIn, conference
- CV screening — приховай ім'я й фото (зменш bias)
- Розмаїтість у interview panel (різні жінки, різні культури)
- Питай про障礎 accessibility: "Чи потрібна вам гнучка робоча часи?"

## Технічні стандарти в команді

### Як запроваджувати без опору

**ПОГАНО (зверху вниз):**
```
"З понеділка все PR повинні мати 100% code coverage.
Це REGEL. Або merge не дам."
```
Результат: люди обходять правила, пишуть фіктивні тести.

**ДОБРЕ (з командою):**
```
1. Зустрічся — обговори цю ідею:
   "Чому вам здається, важливо мати тести?
    Які раніше трапилась issues без тестів?"

2. Спільно вирішіть:
   "80% coverage? 90%? Як це буде виглядати?"

3. Документуйте (RFC або ADR):
   "Коли писати unit vs integration test
    Які inструменти використовуємо"

4. Вводьте поступово:
   "На новому код базі (new features) требув 90%.
    На старому коді можемо 60%."
```

### RFC (Request for Comments) процес

```
1. Одна людина пише документ:
   "Ми переходимо на TypeScript"
   
2. Ділиться у Slack/Google Docs (3-5 днів обговорення)

3. Команда дає feedback:
   - "Це займе скільки часу?"
   - "А як мы тестуємо?"
   - "Чи це обов'язково для всіх?"

4. Автор адаптує документ

5. APPROVED → запроваджуємо
```

### ADR (Architecture Decision Records)

Коротка документація ЧОМУ ви прийняли рішення.

```markdown
# ADR-001: Використання Redis для кешування

## Context
Ми маємо ~50k запросів на /users endpoint в день,
кожен займає 500ms. DB не масштабується.

## Decision
Кешувати результаті в Redis з TTL 10 хвилин.

## Alternatives
1. DB replication (занадто дорого)
2. GraphQL caching (потребує переписування API)

## Consequences
+ 90% reduction у latency (/users тепер 50ms)
+ Потребуємо Redis instance (додатковий сервіс)
- Складність cache invalidation при updates
```

### Code pairing як метод

Раз на тиждень: один юніор + один senior сідають разом на 1 годину.

```
- Senior пише, Junior спостерігає й питає
- Потім вони міняються ролями

Результат:
✓ Юніор вчиться реальним паттернам (не лише code review)
✓ Senior переглядає свої навички / вчить краще пояснювати
✓ Knowledge transfer швидше
```

## Менторинг під час remote роботи

### Що змінюється

**Synchronous став важливіший**
```
В офісі: "Ей, давай на хвилинку!"
Remote:  Потрібна 1:1 планована сесія

Це значить більш формальні 1:1 (але більш структурованих)
```

**Спостереження за сигналами сложніше**
```
В офісі: Ти бачиш, що хтось виглядає стресованим
Remote:  Потрібно активно запитувати "як справи?"

Бернаут у remote можливо помітити пізніше.
```

**Async communication потребує більшої ясності**
```
ПОГАНО:
"Можеш глянути цей PR? 🙃"

ДОБРЕ:
"Прошу переглянути PR #123.
Сфокусуюся на архітектурі (DB schema, caching strategy).
Стиль буде вирішено ESLint.
Твої сигнали потрібні до п'ятниці 5pm."
```

### Best practices для remote менторства

**1. Office hours (асинхронні)**
```
Середа 10am UTC — Junior Hours
"Це мій час для питань. Приходьте у Zoom або write в Slack.
Я буду доступний 60 хвилин."
```

**2. Асинхронні録画 recordings**
```
"Як налаштувати Git Hooks у проекті"
→ 10-хвилинне відео на YouTube (private)
Junior може переглянути в своєму часові zona.
```

**3. Documentation як культура**
```
Задокументуй процеси вичерпно:
- Як запустити dev server
- Як писати migration
- Як читати логи в production
- Як відкласти SQL query

Це замінює "швидкий вопрос у коридорі".
```

**4. Slack як асинхронний паттерн**
```
Юніор: "Як я повинен писати цей endpoint?"

ПОТІМ (30 хв або більше):
Senior: "Ось ідея з кодом, посилання на подібний endpoint,
і коментари чому. Дай мені знати, якщо питання."
```

**5. Pairing все ще важливо, але запланований**
```
Вторник 2pm UTC — Pairing Session
Junior розповідає про перепон, вони籍 разом його развязуют.
```

## Практичний чекліст для нового Tech Lead

### Week 1: Встановити контакт

- [ ] Провести 1:1 із кожним у команді (30 хв)
- [ ] Запитати: "Який найбільший pain point прямо зараз?"
- [ ] Організувати team retro (чути від усіх)
- [ ] Обговорити з попереднім lead: "Що потребує уваги?"

### Місяць 1: Зрозуміти архітектуру

- [ ] Читати код: 3 основні модулі (2-3 години)
- [ ] Запустити проект локально, оновити README якщо потрібно
- [ ] Визначити потенційні bottlenecks (slow queries, memory leaks)
- [ ] Написати 1-2 документи: Architecture Overview + на найменш 3 ADR

### Місяць 1-3: Підготувати менторинг

- [ ] Встановити регулярні 1:1 із junior-ами (кожен > 1 раз на тиждень)
- [ ] Організувати знайомство:團隊團隊 presentations (хто-що робить)
- [ ] Поставити першу задачу junior-ам (щось маленьке й безпечне)
- [ ] Учаснути в code review із junior-ами (показувати процес)

### Місяці 1-3: Культура й стандарти

- [ ] Оцінити поточне: code review, testing, deployment процесс
- [ ] Обговорити з командою: "Що нам потрібно поліпшити?"
- [ ] Запропонувати RFC (1-2 зміни, яких потребує команда)
- [ ] Розпочати знаменити code pairing сесії (1 раз на тиждень)

### Місяці 2-6: Розвиток команди

- [ ] Визначити, який junior готовий до більш складних задач
- [ ] Організувати внутрішню presentation: кожен опаклює одну технічну тему
- [ ] Провести першу серйозну hiring loop (найми одного розробника)
- [ ] Переглянути: як виглядають знаки бернаута? Чи все ОК в команді?

### Ongoing (кожнесяч)

- [ ] Своя 1:1 з manager (feedback і стратегія)
- [ ] Відкриття для feedback від команди (retro або анонімний survey)
- [ ] Читання однієї технічної книги (або статті) про лідерство
- [ ] Прості ніо: рефакторинг / tech debt 10-15% спринту
