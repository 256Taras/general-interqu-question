# Behavioral питання для Tech Lead — STAR метод

## Як використовувати STAR метод

Кожна відповідь має містити:
- **S (Situation)** — контекст, ситуація
- **T (Task)** — твоя задача/відповідальність
- **A (Action)** — що конкретно ти зробив
- **R (Result)** — результат, метрики, висновки

---

## Розкажіть про конфлікт у команді та як ви його вирішили

**S:** Два senior розробники в моїй команді мали протилежні погляди на архітектуру нового сервісу — один наполягав на мікросервісах, інший на модульному моноліті. Дискусія стала емоційною і блокувала прогрес протягом тижня.

**T:** Як tech lead, мені потрібно було розблокувати команду, прийняти обґрунтоване рішення і зберегти здорову атмосферу.

**A:**
1. Провів окремі 1-on-1 з кожним, щоб зрозуміти їхні аргументи без тиску іншого
2. Організував структуровану дискусію з чіткими критеріями оцінки (масштаб, строки, команда, maintenance)
3. Попросив кожного написати ADR (Architecture Decision Record) зі своїм підходом
4. Запросив третю сторону (архітекта з іншої команди) для незалежної оцінки
5. Прийняв рішення на основі критеріїв і пояснив, чому саме так

**R:** Обрали модульний моноліт (відповідав масштабу проекту і розміру команди). Обидва розробники прийняли рішення, бо процес був прозорим. Цей ADR став шаблоном для майбутніх рішень. Проект доставили вчасно.

## Як ви мотивуєте команду?

**S:** Після довгого проекту (6 місяців) без видимих результатів для кінцевих користувачів (інфраструктурний рефакторинг), команда втратила мотивацію. Velocity впала на 30%.

**T:** Повернути мотивацію та показати команді цінність їхньої роботи.

**A:**
1. Розбив великий проект на менші milestones з чіткими "перемогами"
2. Ввів demo sessions щотижня — команда бачила прогрес
3. Показав метрики "до і після" — час деплою зменшився з 45 до 5 хвилин
4. Організував celebration після кожного milestone
5. Дав кожному можливість обрати наступну задачу за інтересом
6. Публічно визнавав внесок кожного на ретроспективі

**R:** Velocity повернулась до нормального рівня за 2 спринти. Один з розробників сказав, що це був найцікавіший проект, коли побачив реальний вплив своєї роботи.

## Розкажіть про невдалий проект і що ви з цього винесли

**S:** Мігрували monolith на мікросервіси. Запланували повну міграцію за 3 місяці для 5 сервісів.

**T:** Як tech lead відповідав за технічне планування, розподіл задач і delivery.

**A:** На початку ми:
- Недооцінили складність data migration
- Не зробили proof of concept перед повною міграцією
- Не врахували зміни у процесі розробки нових фіч паралельно
- Через 2 місяці зрозуміли, що тільки 2 з 5 сервісів готові

**Що зробив після усвідомлення проблеми:**
1. Зупинив "поїзд" — чесно повідомив stakeholders про ситуацію
2. Переглянув план — деплоїмо 2 готових сервісі, решту переносимо
3. Ввів Strangler Fig pattern замість big bang міграції
4. Додав phase gates — кожен сервіс окремо з чіткими критеріями

**R:** Деплойнули 2 сервіси вчасно (reduced scope). Решту 3 — протягом наступних 4 місяців. Головний урок: ніколи не планувати big bang міграцію. Incremental delivery — завжди.

## Як ви приймаєте технічні рішення?

**Мій framework для прийняття рішень:**

1. **Зібрати контекст:** які вимоги, обмеження, строки, розмір команди?
2. **Визначити варіанти:** мінімум 2-3 альтернативи
3. **Оцінити за критеріями:**
   - Відповідність вимогам
   - Складність реалізації
   - Maintenance cost
   - Знання команди
   - Масштабованість
   - Ризики
4. **Обговорити з командою:** RFC або ADR для важливих рішень
5. **Прийняти рішення і задокументувати:** ADR з reasoning
6. **Визначити точку перегляду:** коли перевіримо чи рішення правильне

**Приклад:** Обирали ORM для нового проекту. TypeORM vs Prisma vs Drizzle. Критерії: TypeScript підтримка, продуктивність, learning curve, активність community. Кожен розробник спробував одну ORM на прототипі. Обрали Drizzle — найкращий type safety, SQL-like синтаксис підходив команді з SQL досвідом.

## Як ви проводите Code Review?

**Мої принципи:**

1. **Автоматизувати все, що можна** — linters, formatters, type checks у CI
2. **Фокус на архітектурі**, не на стилі — стиль має забезпечувати tooling
3. **Задавати питання**, а не давати команди — "Чи розглядав ти варіант з...?" замість "Зроби по-іншому"
4. **Хвалити хороший код** — не тільки знаходити проблеми
5. **Чіткі критерії для approve:**
   - Код робить те, що задекларовано
   - Немає security вразливостей
   - Покритий тестами (якщо це критичний path)
   - Зрозумілий без коментарів (або з необхідними коментарями)
6. **Розмір PR** — максимум 400 рядків, інакше розбити
7. **Швидкість** — review протягом 4 годин (не блокувати команду)

**Що шукаю:**
- Потенційні баги і edge cases
- Performance issues
- Security vulnerabilities
- Зрозумілість і maintainability
- Чи дотримується архітектурних паттернів проекту

## Як ви працюєте з дедлайнами?

**S:** Product manager повідомив, що великий клієнт потребує нову фічу через 3 тижні. Моя оцінка — 5-6 тижнів при повному scope.

**T:** Доставити максимальну цінність в заданий timeframe.

**A:**
1. Розбив фічу на must-have / nice-to-have / out-of-scope
2. Погодив з PM reduced scope (must-have = 3 тижні)
3. Створив детальний plan з daily checkpoints
4. Прибрав impediments для команди (meetings, code reviews робив сам швидко)
5. Паралелізував роботу де можливо
6. Щоденний 10-хв sync замість одного великого weekly

**R:** Доставили must-have вчасно. Nice-to-have — через 2 тижні після. Клієнт задоволений, бо основна потреба закрита.

**Принципи:**
- Ніколи не обіцяй те, що не можеш виконати
- Краще зменшити scope, ніж якість
- Комунікуй ризики рано, а не коли вже запізно
- "Ні" із запропонованою альтернативою — це не "ні"

## Як ви менторите junior розробників?

**Мій підхід:**

1. **Onboarding plan** — чіткий план на перші 30/60/90 днів
2. **Pair programming** — 2-3 рази на тиждень перший місяць
3. **Поступове збільшення складності задач:**
   - Тиждень 1-2: bug fixes, малі tasks
   - Тиждень 3-4: нові фічі з guidance
   - Місяць 2-3: самостійні задачі з review
4. **Code review як навчання** — пояснюю чому, а не тільки що
5. **Безпечне середовище** — помилки це нормально, головне вчитись
6. **1-on-1 щотижня** — обговорення прогресу, питань, цілей
7. **Рекомендую ресурси** — книги, курси під конкретні потреби

**Важливо:** не робити за них, а направляти. Ставити питання, які допомагають знайти відповідь самостійно.

## Як ви балансуєте технічний борг та нові фічі?

**Мій підхід — правило 20/80:**
- 80% capacity на нові фічі та бізнес-задачі
- 20% на технічний борг та покращення

**Як пріоритизую tech debt:**
1. **Критичний** (security, data loss risk) — фіксимо одразу
2. **Блокуючий** (заважає розробці нових фіч) — наступний спринт
3. **Важливий** (degrading performance, DX) — планую в roadmap
4. **Бажаний** (refactoring для чистоти) — якщо є час

**Як продаю tech debt бізнесу:**
- Не кажу "треба зробити refactoring" (бізнес не розуміє)
- Кажу "якщо ми зробимо X, швидкість розробки нових фіч зросте на Y%"
- Показую метрики: час на деплой, кількість багів, час на onboarding

## Як ви комунікуєте з нетехнічними стейкхолдерами?

**Принципи:**

1. **Говори мовою бізнесу**, не технічною
   - ❌ "Нам потрібно мігрувати на PostgreSQL 16 для кращої підтримки logical replication"
   - ✅ "Оновлення бази даних дасть нам можливість обробляти в 2 рази більше замовлень"

2. **Візуалізуй** — діаграми, скріншоти, прототипи замість тексту

3. **Пропонуй варіанти** з trade-offs
   - Варіант A: швидше, але менше можливостей
   - Варіант B: повільніше, але повний функціонал

4. **Чесно комунікуй ризики** — рано, а не коли вже проблема

5. **Не перевантажуй деталями** — якщо запитають — розкажеш

## Як ви оцінюєте задачі та плануєте спринти?

**Оцінка задач:**
1. **Planning Poker** з командою (не тільки lead оцінює)
2. **Story Points** для відносної складності (не години!)
3. **Spike tasks** для невизначених задач (timebox 2-4 години на дослідження)
4. **Розбивання великих задач** — якщо > 8 SP, розбити на менші
5. **Історичні дані** — velocity за останні 3-5 спринтів

**Планування спринту:**
1. Цілі спринту визначає Product Owner
2. Як tech lead забезпечую технічну feasibility
3. Команда сама обирає об'єм роботи (не нав'язую)
4. Завжди залишаю 10-15% buffer для непередбачених задач
5. Після планування — кожен знає свої задачі та залежності

**Помилки яких уникаю:**
- Планувати на 100% capacity (завжди буде щось незаплановане)
- Ігнорувати "нецікаві" задачі (bugs, tech debt)
- Не враховувати відпустки, свята, мітинги

## Розкажіть про ситуацію, коли ви не погоджувались з рішенням керівництва

**S:** CTO вирішив використовувати нову, ще не стабільну технологію для критичного проекту з жорстким дедлайном.

**T:** Висловити свою позицію професійно і або переконати, або підтримати рішення.

**A:**
1. Зібрав конкретні факти: відкриті issues, breaking changes в останніх релізах, відсутність production case studies
2. Підготував альтернативу з порівнянням ризиків
3. Презентував CTO: "Я розумію бажання використати нову технологію. Ось мої concerns та альтернативний план"
4. Запропонував компроміс: використати нову технологію для некритичного компонента як proof of concept

**R:** CTO погодився на компроміс. Proof of concept показав кілька критичних проблем. Основний проект зробили на перевіреній технології, а нову — впровадили в наступному кварталі, коли вона стабілізувалась.

**Урок:** "Disagree and commit" — якщо рішення прийняте і я не зміг переконати, я підтримую його на 100%. Але до прийняття рішення — моя відповідальність висловити concerns.

## Як ви впроваджуєте зміни в процесах команди?

**S:** Команда не робила code review — кожен мерджив свій код напряму.

**A:**
1. **Не нав'язував** — пояснив проблему: "За останній місяць ми мали 5 інцидентів, 3 з яких могли бути помічені на review"
2. **Почав з себе** — попросив ревьюїти мій код першим
3. **Зробив процес легким** — малі PR, чіткий checklist, автоматизація (CI checks)
4. **Поступово** — спочатку тільки для critical path, потім для всього
5. **Відзначав позитивні результати** — "Цей баг знайшли на review, а не в production!"

**R:** Через місяць вся команда робила review. Кількість production багів зменшилась на 40%. Команда самі стали advocates процесу.

**Принцип:** Люди приймають зміни, які вони розуміють і в яких бачать цінність. Примус не працює.
