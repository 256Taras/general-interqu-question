# Technical Leadership — Питання для інтерв'ю

## Як ви обираєте технології для нового проекту?

### Мій framework для вибору технологій

**Критерії оцінки:**
1. **Відповідність задачі** — чи вирішує технологія конкретну проблему?
2. **Зрілість** — production-ready? Стабільний API? Довгостроковий support?
3. **Знання команди** — команда вже знає, чи потрібен onboarding?
4. **Community та екосистема** — активна community? Достатньо бібліотек?
5. **Performance** — чи задовольняє вимоги до продуктивності?
6. **Hiring** — чи зможемо найняти людей з цим досвідом?
7. **Total Cost of Ownership** — не тільки ліцензія, а й maintenance, hosting, training

**Процес:**
1. Визначити вимоги (функціональні та нефункціональні)
2. Shortlist 2-3 варіанти
3. Proof of Concept для кожного (timeboxed — 1-2 дні)
4. Порівняння за критеріями (таблиця)
5. Обговорення з командою
6. Рішення + ADR (Architecture Decision Record)

**Приклад:** Обирали між Fastify та Express для нового проекту. Критерії: performance, TypeScript підтримка, schema validation, екосистема. Fastify переміг по performance (2x), нативна TypeBox validation, plugin system. Express мав більшу екосистему, але для нашого випадку Fastify plugins були достатні.

## Як ви впроваджуєте нові практики в команді?

### Поетапний підхід

**1. Покажи проблему (не рішення)**
- Збери дані: скільки багів, час деплою, feedback розробників
- Покажи конкретні приклади проблем

**2. Запропонуй і обговори**
- Презентуй рішення команді (не нав'язуй)
- Дай команді можливість запропонувати альтернативи
- Голосування або консенсус

**3. Pilot**
- Впроваджуй на одному проекті/модулі
- Збирай feedback
- Коригуй підхід

**4. Масштабуй**
- Після успішного pilot — розширюй на всю команду
- Документуй процес
- Навчи тих, хто не брав участь у pilot

**5. Ітеруй**
- Регулярно збирай feedback
- Готовий відмовитись якщо не працює

**Приклад:** Впроваджував automated testing. Почав з одного модуля, написав перші тести сам. Показав команді, як тест знайшов баг до production. Через місяць вся команда писала тести добровільно.

## Як ви працюєте з технічним боргом?

### Класифікація tech debt

**За серйозністю:**
- **Critical** — security vulnerabilities, data loss risks → фіксити одразу
- **High** — блокує development, significant performance issues → наступний спринт
- **Medium** — ускладнює maintenance, DX → планувати в roadmap
- **Low** — code style, minor refactoring → Boy Scout Rule

**За типом:**
- **Deliberate** — свідомий вибір (MVP, deadline) → задокументований, запланований
- **Accidental** — недостатні знання, зміна вимог → потребує refactoring
- **Bit rot** — застарілі залежності, deprecated API → регулярне оновлення

### Стратегії управління

1. **Tech Debt Register** — список всього відомого tech debt з пріоритетами
2. **20% правило** — 20% capacity кожного спринту на tech debt
3. **Boy Scout Rule** — залишай код чистішим ніж знайшов
4. **Tech Debt Sprints** — раз на квартал спринт повністю на tech debt
5. **Метрики** — відстежувати tech debt score (SonarQube, CodeClimate)

### Як "продати" tech debt бізнесу

Не кажи: "Треба рефакторити, бо код поганий"
Кажи:
- "Якщо зробимо X, час розробки нових фіч зменшиться на 30%"
- "Без оновлення Y, ми маємо ризик security breach"
- "Інвестиція в Z дозволить нам деплоїти 5 разів на день замість 1 разу на тиждень"

## Як ви приймаєте рішення "build vs buy"?

### Framework для рішення

| Критерій | Build | Buy/Use existing |
|---|---|---|
| Core business logic | ✅ Завжди build | ❌ |
| Commodity (auth, email, payments) | ❌ | ✅ Є перевірені рішення |
| Конкурентна перевага | ✅ Кастомне рішення | ❌ |
| Time-to-market критичний | ❌ | ✅ Швидше |
| Специфічні вимоги | ✅ Повний контроль | ❌ Може не підходити |
| Довгостроковий maintenance | Потребує команди | Vendor підтримує |

### Приклади

**Build:** Recommendation engine для e-commerce — це конкурентна перевага, специфічна для бізнесу.

**Buy:** Payment processing — використай Stripe. Auth — використай Auth0/Clerk. Email — використай SendGrid. Не винаходь велосипед.

**Hybrid:** Використай бібліотеку і обгорни в свою абстракцію для можливої заміни в майбутньому.

## Як ви забезпечуєте якість коду в команді?

### Автоматизація (перший рівень)
- **Linting** — ESLint з strict конфігурацією
- **Formatting** — Prettier (жодних дискусій про стиль)
- **Type checking** — TypeScript strict mode
- **Tests** — обов'язковий мінімум coverage для CI
- **Security scanning** — npm audit, Snyk
- **Pre-commit hooks** — lint-staged + husky

### Code Review (другий рівень)
- Максимальний розмір PR — 400 рядків
- Мінімум 1 approve для merge
- Автоматичні CI checks перед review
- Review guidelines документовані

### Архітектурні guardrails (третій рівень)
- ADR для важливих рішень
- Coding standards документовані
- Архітектурні тести (dependency rules)
- Регулярні tech talks для обміну знаннями

### Культура (четвертий рівень)
- Pair programming для складних задач
- Blameless culture — помилки це можливість навчитись
- Внутрішні demo і tech talks
- Книжковий клуб

## Як ви плануєте архітектуру?

### Процес

**1. Розуміння вимог (1-2 дні)**
- Функціональні вимоги від product
- Нефункціональні вимоги (performance, scale, security)
- Обмеження (deadline, budget, team size)

**2. Дослідження (1-3 дні)**
- Вивчити існуючу систему
- Benchmarks та PoC для невідомих компонентів
- Вивчити як інші вирішують подібні задачі

**3. Проектування (2-5 днів)**
- High-level architecture diagram
- Data flow diagrams
- API contracts
- Database schema
- Визначення trade-offs

**4. Review (1-2 дні)**
- RFC документ для команди
- Збір feedback
- Коригування

**5. Документація**
- ADR для кожного важливого рішення
- Architecture overview diagram
- Component diagram
- Deployment diagram

### Принципи архітектури
- **YAGNI** — не проектуй на 5 років вперед
- **KISS** — найпростіше рішення, яке працює
- **Incremental** — можна доставляти поступово
- **Evolvable** — легко змінювати при нових вимогах

## Як ви проводите технічні ретроспективи?

### Формат (60 хвилин)

**1. Підготовка (до зустрічі)**
- Кожен записує: що пішло добре, що погано, що покращити
- Анонімний збір feedback (Miro, FigJam)

**2. Огляд попередніх action items (5 хв)**
- Що зробили з минулої ретроспективи?
- Що не зробили і чому?

**3. Обговорення (40 хв)**
- Групування схожих тем
- Голосування за пріоритети (dot voting)
- Глибоке обговорення top 3 тем

**4. Action items (15 хв)**
- Конкретні дії з відповідальними та дедлайнами
- Максимум 3 action items (більше — не виконаються)

### Типи ретроспектив
- **Sprint retro** — кожні 2 тижні, фокус на процесі
- **Project retro** — після завершення проекту, фокус на outcomes
- **Incident retro (postmortem)** — після інциденту, blameless

## Як ви працюєте з legacy кодом?

### Стратегія

**1. Зрозуміти перед тим як змінювати**
- Читай код, не переписуй одразу
- Знайди відповідальних (git blame)
- Зрозумій чому код такий (можливо, є вагома причина)

**2. Стабілізувати**
- Додати тести для критичних paths
- Характеризаційні тести (фіксують поточну поведінку)
- Моніторинг та alerting

**3. Поступово покращувати**
- Strangler Fig Pattern — нові фічі на новому коді
- Boy Scout Rule — покращуй те, до чого торкаєшся
- Extract і Replace — виділяй компоненти і замінюй

**4. Не переписувати з нуля**
- "The Big Rewrite" майже завжди провалюється
- Поступова міграція набагато безпечніша
- Зберігай працюючу систему поки мігруєш

### Типові помилки з legacy
- Переписати все з нуля ("тут все погано, починаємо заново")
- Не писати тести перед рефакторингом
- Не розуміти бізнес-логіку перед зміною
- Змінювати все одразу замість поступових кроків

## ADR (Architecture Decision Records) — що це і навіщо?

### Формат ADR

```markdown
# ADR-001: Вибір ORM для проекту

## Статус
Accepted (2024-01-15)

## Контекст
Потрібно обрати ORM для нового Node.js проекту з PostgreSQL.
Команда з 5 розробників, всі мають досвід з SQL.

## Варіанти
1. TypeORM — Active Record + Data Mapper, великий community
2. Prisma — Schema-first, type-safe, свій migration tool
3. Drizzle — SQL-like синтаксис, найкращий type safety

## Рішення
Обираємо Drizzle ORM.

## Обґрунтування
- SQL-like синтаксис підходить команді з SQL досвідом
- Найкращий TypeScript type inference
- Менший overhead ніж Prisma (немає Rust engine)
- Активна розробка та зростаюча community

## Наслідки
- Потрібен час на вивчення Drizzle API (1-2 дні)
- Менша екосистема плагінів ніж у TypeORM
- Міграції через drizzle-kit

## Учасники
@team-lead, @senior-dev-1, @senior-dev-2
```

### Навіщо ADR?
- **Історія рішень** — через рік зрозумієш чому обрали саме так
- **Onboarding** — нові люди швидше розуміють архітектуру
- **Уникнення повторних дискусій** — рішення задокументоване
- **Accountability** — зрозуміло, хто і коли прийняв рішення

## Як ви документуєте архітектурні рішення?

### Рівні документації

**1. Code level** — self-documenting code + коментарі "чому"
**2. Component level** — README в кожному модулі
**3. System level** — Architecture overview, ADRs, deployment diagrams
**4. Process level** — Runbooks, playbooks, onboarding guides

### Інструменти
- **ADRs** — для рішень (у репозиторії, поруч з кодом)
- **Diagrams** — C4 Model (Context, Container, Component, Code)
- **Wiki** — для процесів і runbooks (Notion, Confluence)
- **OpenAPI/Swagger** — для API документації (автоматично з коду)

### Принципи
- Документація поруч з кодом (не в окремій системі)
- Автоматична де можливо (OpenAPI, TypeDoc)
- Актуальна — якщо застаріла, краще видалити
- Для аудиторії — різна документація для різних людей (dev, ops, business)
